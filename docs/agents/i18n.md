# Internationalization (i18n) reference

Canonical reference for translating OpenPorte. The **`.po` files under `languages/` are the
source of truth** — there is no separate dictionary or build step. The toolchain is WP-CLI
(`wp i18n …`) plus GNU gettext (`msgfmt`). See `AGENTS.md` → "i18n discipline" for the rules
that apply while *writing* code (wrapping strings, `translators:` comments); this document
covers *producing and maintaining* the translations.

## Current state

- **Fully translated:** `de_DE`, `es_ES`, `fr_FR`, `it_IT`, `nl_NL`, `pt_PT`.
- **Stubs (empty `msgstr`, awaiting translation):** the remaining ~23 locales shipped under
  `languages/` (e.g. `zh_CN`, `ja`, `pl_PL`, …). They already carry the full `msgid` set, so a
  translator/LLM only has to fill in the `msgstr` values.

## Prerequisites

- [WP-CLI](https://wp-cli.org/) with the i18n command (`wp i18n`).
- GNU gettext for validation (`msgfmt`). On macOS: `brew install gettext`.

## The toolchain (single source of truth)

```bash
# 1. Regenerate the POT template from the source code
wp i18n make-pot . languages/openporte.pot \
  --domain=openporte \
  --exclude=vendor,local,tests \
  --slug=openporte

# 2. Sync every locale's .po against the new template
#    New strings appear with an empty msgstr; changed strings are marked "#, fuzzy".
wp i18n update-po languages/openporte.pot languages/

# 3. Stamp the release version into the .po headers (replace X.Y.Z).
#    NOTE the trailing \n" — the metadata line MUST keep its newline, or gettext
#    merges Project-Id-Version into the next header field.
sed -i '' \
  's/Project-Id-Version: OpenPorte Spam Protection.*/Project-Id-Version: OpenPorte Spam Protection X.Y.Z\n"/' \
  languages/openporte-*.po
# On Linux, drop the '' after -i.

# 4. Compile the .mo binaries WordPress actually loads
wp i18n make-mo languages/
```

`update-po`/`make-mo` accept a single file as the destination/source if you want to touch only
one locale, e.g. `wp i18n update-po languages/openporte.pot languages/openporte-fr_FR.po`.

### Adding a new language to the list of translations

After the regeneration of the POT template from our toolchain standard workflow, you can insert
the following steps:

```bash
# 2-bis. Create the new language .po file by copying the .pot
cp languages/openporte.pot languages/openporte-*.po
# e.g.: cp languages/openporte.pot languages/openporte-xh_ZA.po

# 2-bis.1. Edit the .po header — at minimum set these fields by either adding or updating them (for plural forms, the (Unicode CLDR plural rules)[https://unicode-org.github.io/cldr-staging/charts/latest/supplemental/language_plural_rules.html] is the authoritative reference):
#    "Language: <language[_territory]>\n"
#    "PO-Revision-Date: <date>\n"
#    "Last-Translator: <name>\n"
#    "Language-Team: <Language_name>\n"
#    "Plural-Forms: nplurals=2; plural=(n != 1);\n"

# 2-bis.2. Translate the strings in the .po file
# (by hand, or feed it to a translator/LLM)
```

## Glossary — terms to preserve verbatim

Never translate these (leave `msgstr` identical to `msgid`, or keep the term inline in a longer
string):

**Brand / plugin / theme names:** OpenPorte, OpenPorte Spam Protection, ALTCHA,
ALTCHA Spam Protection, WordPress / Wordpress, WooCommerce, Contact Form 7, Gravity Forms,
Forminator, Formidable Forms, Elementor Pro Forms, CoBlocks, WPDiscuz, WP Forms, HTML Forms,
Enfold Theme, Custom HTML, Shortcode.

**Technical terms:** CAPTCHA, JavaScript, GDPR, GitHub, REST API, HMAC, PoW / proof-of-work,
PHP, WP-CLI, GPL.

**Always preserve exactly, in every language:**
- URLs (`http://…`, `https://…`).
- Placeholders: `%s`, `%d`, `%1$s`, `%2$s`, … — same set, same order semantics.
- HTML tags: `<a>`, `<b>`, `</a>`, … — same tags, unchanged.

> Some terms have an accepted local form (e.g. *CAPTCHA* may stay, but "proof-of-work" can be
> explained). When in doubt, prefer the English term over an awkward coinage.

## Quality rules

- **Accuracy** — natural and idiomatic in the target language; keep the source's level of formality.
- **Consistency** — translate a given term the same way everywhere; reuse existing choices in the
  same `.po` before inventing new ones.
- **Completeness** — every translatable string gets a `msgstr`; empty `msgstr` should be rare. If
  genuinely unsure, leave the English text rather than guessing.
- **Comments** — keep every `#. translators:` comment. Any string with a placeholder should have one.

## Procedure: refresh a language after code changes

Use when source strings were added/changed (the i18n discipline flagged it in a commit). Example: `fr_FR`.

1. Regenerate the template and sync — steps 1–2 above. New strings land empty; reworded strings
   become `#, fuzzy` (the old translation is kept as a suggestion).
2. Fill the gaps: translate every empty `msgstr`, review each `#, fuzzy` entry, and **remove the
   `fuzzy` flag** once corrected. This is the natural place to hand the file to a local LLM (below).
3. Validate: `msgfmt --check languages/openporte-fr_FR.po -o /dev/null` (0 errors).
4. Compile: `wp i18n make-mo languages/openporte-fr_FR.po languages/openporte-fr_FR.mo`.

## Procedure: add a new language

Example: Simplified Chinese, `zh_CN` (already a stub here; for a brand-new locale, `update-po`
creates the `.po` from the template).

1. Ensure the stub is current: `wp i18n update-po languages/openporte.pot languages/openporte-zh_CN.po`.
2. Translate every `msgstr` (hand it to a local LLM with the prompt below).
3. Validate: `msgfmt --check languages/openporte-zh_CN.po -o /dev/null`.
4. Compile: `wp i18n make-mo languages/openporte-zh_CN.po languages/openporte-zh_CN.mo`.
5. Smoke-test in wp-env (switch the site language and check the OpenPorte settings screen).

## LLM-assisted translation

The LLM edits the `.po` file **directly** — that keeps gettext as the single source of truth and
needs no custom tooling. Open the target `.po`, hand its full contents plus the prompt below to a
local model, then replace the file with the model's output and validate with `msgfmt --check`
(which also verifies placeholders via `--check-format`). Work one locale at a time.

### Prompt template

Replace the template keywords:
- <TARGET LANGUAGE>: the new translation language to be created, in plain English (e.g. Xhosa).
  There is only one instance of that template keyword in the prompt.
- <LOCALE>: Follow the POSIX (IEEE 1003.1) notation, which is `language[_territory]`,
  language being the ISO 639-1 (e.g. xh for Xhosa, or fr for French) and the territory
  being ISO 3166-1 (e.g. ZA for South Africa, or CA for Canada). There are two instances of that
  template keyword in the prompt.

Example locales:
- xh_ZA: Xhosa in South Africa
- fr_CA: French in Canada

---

```
You are a professional software localizer translating a WordPress plugin's gettext catalog into
<TARGET LANGUAGE> (locale <LOCALE>, e.g. Simplified Chinese / zh_CN).

The target file to translate is (and refer to after as target_file.po):
languages/openporte-<LOCALE>.po

Steps — follow exactly, in order:
1. Read the full contents of target_file using your file tool.
2. Produce a translated version of that file, applying the rules below.
3. Overwrite target_file.po with the translated content using your file write/edit
   tool. Do NOT just print the result to the console or chat — the file on disk must be modified.
4. After writing, reply with only: "Translation finished, file target_file.po saved." (replace
   "target_file.po" by the real file name). Do not paste the file content back to me.

Translation rules:
1. Translate ONLY the `msgstr` values. Never alter a `msgid`, comment, reference line, or flag,
   except: when an entry is marked `#, fuzzy`, correct its `msgstr` and DELETE the `fuzzy` flag.
2. Leave these untranslated, exactly as written in the source:
   - Brand/plugin/theme names: OpenPorte, ALTCHA, WordPress, WooCommerce, Contact Form 7,
     Gravity Forms, Forminator, Formidable Forms, Elementor Pro Forms, CoBlocks, WPDiscuz,
     WP Forms, HTML Forms, Enfold Theme, Custom HTML, Shortcode.
   - Technical terms: CAPTCHA, JavaScript, GDPR, GitHub, REST API, HMAC, PoW / proof-of-work,
     PHP, WP-CLI, GPL.
   - All URLs.
3. Preserve every placeholder (%s, %d, %1$s, %2$s, …) — same set and the same argument numbers —
   and every HTML tag (<a>, <b>, …) exactly. Reorder them within a sentence only if the target
   grammar requires it; never add or drop one.
4. Honour each `#. translators:` comment — it explains what the placeholders mean.
5. Be natural and idiomatic; match the source's formality. Keep translations concise so they fit
   in UI controls. Do not leave any `msgstr` empty.
6. Keep the file a syntactically valid gettext .po file — same structure, same entry order, same
   headers, only `msgstr` values (and `fuzzy` flags per rule 1) change.

Do not ask for confirmation before writing the file. Do not summarize the changes in your reply.
Do not stage nor commit.
```

---

After pasting the model's output back into the file:

```bash
msgfmt --check languages/openporte-<LOCALE>.po -o /dev/null   # must report 0 errors
wp i18n make-mo languages/openporte-<LOCALE>.po languages/openporte-<LOCALE>.mo
```

`msgfmt --check` is the gate: it catches malformed syntax and any placeholder mismatch between
`msgid` and `msgstr`. Fix anything it reports before committing.
